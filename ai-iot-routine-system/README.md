# AI·IoT 기반 능동형 루틴 관리 및 피드백 시스템

스마트폰 사용 로그와 IoT 센서 데이터를 결합해 사용자의 실제 행동을 자동으로 인식하고, 루틴 수행 여부를 판단해 실시간 피드백과 AI 기반 성공 예측을 제공하는 능동형 루틴 관리 시스템입니다.

---

## 1. 프로젝트 개요

기존의 루틴 관리 서비스는 사용자가 직접 완료 여부를 기록해야 하는 수동적 방식에 의존하기 때문에, 실제 행동과 계획 사이에 차이가 발생하기 쉽고 기록 자체의 신뢰성도 떨어질 수 있습니다.

이 프로젝트는 이러한 한계를 개선하기 위해, **스마트폰 사용 로그와 IoT 센서 데이터를 결합해 사용자의 실제 행동을 자동으로 인식하고**, 루틴 시간대 안에서 수행 여부를 판단해 **실시간 피드백과 AI 기반 성공 예측**을 제공하는 시스템을 설계·구현한 프로젝트입니다.

시스템은 루틴 설정, 스마트폰 상태 로그, ESP32 센서 로그, 규칙 엔진 상태, 일일 루틴 요약 로그를 중심으로 구성했으며, 이를 바탕으로 자동 판단과 통계 제공이 가능하도록 설계했습니다.

### 기간 / 형태

- **기간:** 2025.11.11 ~ 2025.12.07
- **형태:** 1인 개발 프로젝트

### 한 줄 소개

스마트폰 사용 로그와 ESP32 센서 데이터를 결합해 루틴 수행 여부를 자동 인식하고, 규칙 기반 알림과 AI 성공률 예측을 제공하는 AI·IoT 기반 능동형 루틴 관리 시스템

## 2. 문제 정의

기존의 할 일 관리 앱이나 캘린더 서비스는 사용자가 직접 완료 여부를 기록해야 하기 때문에, 실제 행동과 기록이 불일치할 수 있다는 문제가 있습니다.

특히 루틴 관리에서는 다음과 같은 한계가 있습니다.

- 완료 여부를 기록하지 않으면 데이터가 누락됨
- 실제로 수행하지 않았더라도 임의로 완료 처리할 수 있음
- 단순 기록만으로는 실패 원인을 파악하기 어려움
- 실시간 개입 없이 사후 확인에 그치는 경우가 많음

이 프로젝트는 이를 해결하기 위해 다음 방향으로 설계했습니다.

- **Focus Window 기반 동작**으로 루틴 시간대에만 판단 수행
- **스마트폰 로그 + 조도 + 모션 + BLE**를 결합한 멀티모달 입력 구조 적용
- **규칙 기반 실시간 판단과 AI 기반 예측**을 분리해 즉각 대응과 장기 분석을 동시에 지원
- **객관적 행동 데이터 기반 루틴 로그**를 축적해 통계와 성공률 예측에 활용

## 3. 시스템 아키텍처

시스템은 크게 **Sensing Layer**, **Core Server Layer**, **Action / Feedback Layer**로 구성했습니다.

### 1) Sensing Layer

- **Flutter 모바일 앱**이 스마트폰 화면 상태, 잠금 여부, 사용 시간 등의 정보를 수집
- **ESP32**가 조도(LDR), 움직임(PIR), BLE 신호를 수집
- 수집된 데이터는 각각 백엔드 API로 전송되어 루틴 수행 판단의 입력값으로 사용

### 2) Core Server Layer

- **Raspberry Pi 기반 FastAPI 서버**가 전체 데이터 수집과 판단 로직을 담당
- 스마트폰 상태: `/api/device/state`
- 센서 데이터: `/api/sensors/report`
- BLE 근접 정보: `/api/ble/proximity`
- 현재 루틴 상태 점검: `/api/rules/check-now`
- 사용자 피드백: `/api/rules/feedback`
- AI 성공률 요약: `/api/ml/routine-success/summary`
- 습관 통계 조회: `/api/stats/habits`

### 3) Action / Feedback Layer

- 규칙 엔진이 현재 상태를 **ON_TASK / BORDERLINE / OFF_TASK**로 판단
- 필요 시 앱 푸시 알림을 발송
- 사용자의 피드백을 이후 판단에 반영
- 누적 데이터는 AI 분석과 습관 통계 생성에 활용

### 데이터 저장 구조

데이터베이스는 다음과 같은 엔티티 중심으로 설계했습니다.

- **Routine**: 루틴 이름, 수행 요일, 시작/종료 시각, 허용 오차, 수면 루틴 여부
- **DeviceStateLog**: 스마트폰 화면 상태, 잠금 여부, 당일 사용 시간
- **Esp32SensorLog**: 조도, 움직임, BLE RSSI
- **RoutineLog**: 날짜별 최종 상태, 집중도 점수, 화면 사용 비율
- **RuleState**: 마지막 상태, 마지막 알림 시점, 최근 피드백 상태

또한 **Routine : RoutineLog = 1:N** 관계를 두어, 하나의 루틴에 대해 날짜별 요약 로그가 누적되도록 구성했습니다.

## 5. 담당 역할

- 루틴 규칙 엔진 설계 및 상태 판정 로직 구현

### 구현 의도

이 프로젝트는 단순히 루틴을 기록하는 앱이 아니라, **실제 행동을 인식해 자동으로 수행 여부를 추론하는 구조**를 목표로 했습니다.

이를 위해 모바일 로그와 IoT 센서를 함께 사용하는 **멀티모달 입력 구조**를 설계했고, 루틴 시간대 안에서만 시스템이 동작하도록 구성해 **선택적 관찰 기반 시스템**으로 방향을 잡았습니다.

또한 원천 로그와 일일 요약 로그를 분리해 저장하고, 이렇게 축적된 데이터를 규칙 엔진의 즉시 판단과 AI 기반 성공 확률 예측에 모두 활용할 수 있도록 전체 흐름을 설계했습니다.

## 6. 핵심 설계 포인트

### 1) Focus Window 기반 프라이버시 고려

이 시스템은 24시간 사용자를 감시하는 구조가 아니라, 사용자가 설정한 루틴 시간대에만 데이터를 해석하고 판단하도록 설계했습니다.

이를 통해 다음을 목표로 했습니다.

- 불필요한 데이터 수집 최소화
- 프라이버시 부담 완화
- 루틴 수행 판단에 필요한 시간대에만 집중

### 2) 멀티모달 센싱 기반 판단 정확도 보완

스마트폰 화면이 켜졌다는 이유만으로 반드시 루틴을 이탈했다고 볼 수는 없기 때문에, 단일 입력만으로 판단하지 않고 다음 정보를 함께 고려했습니다.

- 스마트폰 사용률
- 조도 비율
- 움직임 감지 여부
- BLE RSSI 기반 책상 근처 체류 여부

이 구조를 통해 단순 화면 사용 여부보다 더 맥락적인 판단이 가능하도록 했습니다.

### 3) 슬라이딩 윈도우 기반 상태 판정

순간적인 행동을 곧바로 이탈로 판단하지 않기 위해, 최근 **10분간의 데이터 흐름**을 기준으로 상태를 판정하는 슬라이딩 윈도우 방식을 적용했습니다.

- **OFF_TASK:** 스마트폰 사용률 70% 이상
- **BORDERLINE:** 스마트폰 사용률 30% 이상
- **ON_TASK:** 스마트폰 사용률 30% 미만

이 방식은 일시적 행동과 지속적 이탈을 구분하기 위한 핵심 판단 로직이었습니다.

### 4) 규칙 기반 실시간 개입 + AI 기반 장기 분석

즉각적인 알림은 규칙 엔진이 담당하고, 장기적인 성공 가능성 예측은 Random Forest 모델이 담당하도록 역할을 분리했습니다.

- 현재 순간에는 규칙 기반 판단으로 즉시 개입
- 누적 데이터에 대해서는 AI가 성공 확률을 예측해 사전 피드백 제공

## 7. 트러블슈팅

### 1) 단일 센서 신호만으로는 루틴 수행 여부를 정확히 판단하기 어려움

#### 문제
초기에는 스마트폰 화면 상태만으로 루틴 수행 여부를 판단하려 했지만, 짧게 시간을 확인하거나 메시지를 보는 행동도 모두 루틴 이탈처럼 보이는 문제가 있었습니다.

#### 원인
스마트폰 사용이라는 단일 신호만으로는 사용자의 실제 의도와 현재 맥락을 충분히 설명하기 어려웠습니다.

#### 해결
판단 기준을 단일 이벤트가 아니라 **최근 10분간의 누적 비율**로 바꾸고, 스마트폰 로그 외에 **조도, 모션, BLE RSSI** 데이터를 함께 반영하는 구조로 확장했습니다.

#### 배운 점
행동 인식 시스템에서는 센서 하나의 정확도보다 **여러 신호를 어떻게 조합해 해석하느냐**가 더 중요하다는 점을 배웠습니다.

### 2) 실시간 알림이 잦아질 경우 사용자 피로도가 높아질 수 있음

#### 문제
규칙 기반으로 현재 상태를 자주 점검하다 보면, 짧은 이탈이나 경계 상태에서도 반복적으로 알림이 발생해 사용자 경험이 나빠질 수 있었습니다.

#### 원인
실시간 개입 자체는 중요하지만, 사용자 맥락을 반영하지 않은 반복 알림은 오히려 시스템 신뢰를 떨어뜨릴 수 있었습니다.

#### 해결
사용자 피드백 API를 두어 알림에 대해 “수행 중”과 같은 응답을 받을 수 있도록 했고, 이 값을 **RuleState**에 저장해 이후 알림 억제 판단에 활용할 수 있도록 설계했습니다.

#### 배운 점
실시간 피드백 시스템은 단순히 빨리 반응하는 것보다 **적절한 빈도로 개입하는 것**이 더 중요하다는 점을 확인했습니다.

### 3) 행동 로그는 많지만 바로 분석 가능한 형태로 쓰기 어려움

#### 문제
스마트폰 로그와 센서 로그는 시계열 데이터로 계속 쌓이지만, 이를 그대로는 루틴 성공률 분석이나 일별 통계에 활용하기 어려웠습니다.

#### 원인
원천 로그는 세밀한 반면, AI 분석과 사용자에게 보여줄 통계는 루틴 단위의 요약 형태가 더 적합했기 때문입니다.

#### 해결
원천 데이터는 각각 로그 테이블에 저장하고, 별도로 **RoutineLog**를 두어 날짜 단위로 상태, 집중도 점수, 화면 사용 비율 등을 집계할 수 있도록 설계했습니다.

#### 배운 점
행동 인식 시스템에서는 데이터 수집만 중요한 것이 아니라, **분석 가능한 단위로 재구성하는 저장 구조**까지 함께 설계해야 한다는 점을 배웠습니다.

## 9. 결과 및 한계

### 결과

- 루틴 관리 문제를 단순 체크리스트가 아니라 **행동 인식 기반 문제**로 재정의
- 스마트폰 로그와 IoT 센서를 결합한 **능동형 루틴 관리 시스템 구조** 설계 및 구현
- 실시간 상태 판단, 피드백 반영, 일일 로그 집계, 성공률 예측으로 이어지는 전체 흐름 구성
- 모바일, 임베디드, 백엔드, AI 요소를 하나의 서비스 흐름 안에서 연결하는 경험 확보

### 한계

- 장기간 사용자 데이터가 충분하지 않아 AI 모델의 일반화 성능 검증에는 한계가 있었음
- BLE RSSI와 센서 값은 환경에 따라 편차가 커 실사용 환경마다 보정이 필요할 수 있음
- 루틴 종류에 따라 판단 기준이 달라질 수 있어 현재 임계값 기반 규칙의 개인화 수준은 제한적임
- Google Calendar 연동은 구조적으로 고려했지만 완전한 운영 수준 통합까지는 추가 구현이 필요함

### 개선 방향

- 사용자별 루틴 특성에 맞춘 **개인화 임계값 자동 조정**
- 장기 데이터 축적을 통한 **AI 예측 정확도 개선**
- 센서 노이즈 보정을 위한 전처리 강화
- 알림 빈도 및 시점 최적화를 위한 정책 고도화
- 외부 일정 데이터와의 연동 강화로 계획-행동 비교 정밀도 향상
